---

## 2. 总体架构

系统采用 **前后端分离** 架构：

- **前端**：浏览器中的单页应用（SPA），通过 HTTPS/HTTP 调用后端 RESTful API。  
- **后端**：Node.js + Express 提供 API、鉴权、业务规则与外部 AI 调用。  
- **数据库**：MySQL 8.x 持久化用户、排班、预约、就诊记录、公告与审计等数据。

典型请求链路：浏览器 → Express 路由 → 控制器 → 服务层（如 AI）→ MySQL → JSON 响应。

## 5. 数据库设计概要

核心实体关系简述：

- **users**：系统账号与角色（admin / staff / doctor / patient）。  
- **department**、**doctor**：科室与医生档案；医生与用户一对一关联。  
- **schedule**：医生在某日某时段的号源与配额。  
- **appointment**：患者对某号源的预约订单及状态。  
- **medical_record**：与预约关联的就诊记录（诊断、医嘱等）。  
- **inquiry**：AI 分诊问询记录（便于审计与展示）。  
- **announcement**：公告。  
- **system_settings**：可配置项（如 AI 地址与模型）。  
- **audit_log**（若项目版本已建表）：关键操作审计。

外键与唯一约束用于保证数据一致性（如单预约单条就诊记录）。

---

## 6. 后端模块结构（逻辑）

- **routes**：注册 API 路径，区分需登录与公开接口。  
- **controllers**：处理请求参数、调用服务、返回统一 JSON。  
- **middleware**：JWT 鉴权、安全头、限流等。  
- **services**：AI 调用、评分等跨控制器逻辑。  
- **db**：连接池、启动时建表或迁移逻辑（以项目实现为准）。

---

## 7. 安全设计要点

- **鉴权**：基于 JWT，敏感接口校验角色。  
- **登录限流**：降低密码爆破风险。  
- **密码存储**：使用 bcrypt 等单向哈希（以代码为准）。  
- **HTTP 安全**：安全相关中间件（如限流、CORS 白名单在生产环境由 `NODE_ENV` 控制）。  
- **审计与追踪**：关键写操作记入审计表；请求维度 `request_id` 便于排障。  
- **AI 超时**：避免长时间挂死工作线程。

---

## 8. 前端工程要点

- **路由**：按角色展示不同菜单与页面。  
- **HTTP 封装**：统一 baseURL、Token 注入、错误处理；对 GET 弱网重试（若已启用）。  
- **构建产物**：生产构建输出于 `client/dist`，由 `vite preview` 或静态服务器托管。

---

## 9. AI 分诊实现思路

- 将用户症状与安全上下文封装为提示词，请求兼容 OpenAI 的 Chat Completions 接口。  
- 解析模型返回 JSON 或结构化文本，映射为「推荐科室、置信度、建议」等字段落库。  
- 支持在后台切换 Base URL 与模型名，以便接入校内网关或本地 Ollama。

## 12. 静态原型说明

单文件 `门诊预约系统-静态原型.html` 使用哈希路由模拟多页，**不依赖后端**。用于作业展示界面与交互流程，与主系统代码库分离维护。
